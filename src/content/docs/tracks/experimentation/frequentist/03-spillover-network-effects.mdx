A foundational assumption of any classical A/B test is the **Stable Unit Treatment Value Assumption (SUTVA)**.

SUTVA requires that the potential outcomes for any user are completely unaffected by the treatment assignment of any other user. If assigning User A to the Treatment group alters the behavior of User B in the Control group, SUTVA is violated.

This violation is known as **Interference** or **Spillover**, and it fundamentally destroys the validity of standard [Difference-in-Means estimation](/gradient-lectures/tracks/experimentation/frequentist/02-core-estimation/) because the Control group is no longer a valid counterfactual.

## How Spillover Occurs

Spillover typically manifests in two distinct environments, each biasing results in opposite directions.

### 1. Network Effects (Social Platforms)

In social networks or communication apps, users interact directly with each other.

**Scenario:** A new frictionless photo-sharing feature is tested. User A (Treatment) gets the feature and sends 50% more photos to their friends. User B (Control) receives these extra photos and opens the app more frequently to view and reply to them.

**The bias:** User B's engagement metrics artificially inflate due to User A's treatment. Because the Control group's baseline is dragged upward alongside the Treatment group, the Difference-in-Means will severely **underestimate** the true treatment effect.

### 2. Cannibalization (Two-Sided Marketplaces)

In marketplaces like Uber or Airbnb, users compete for a finite pool of resources.

**Scenario:** A new routing algorithm for drivers is tested. Driver A (Treatment) gets the algorithm and finds rides 10% faster, picking up more riders. This shrinks the available rider pool for Driver B (Control), whose metrics drop.

**The bias:** The Control group's baseline is dragged downward, causing the Difference-in-Means to **overestimate** the true treatment effect.

## Robust Mitigation Strategies

When severe spillover is expected, standard user-level randomization is mathematically invalid. We must change the **unit of randomization**.

### 1. Graph Cluster & Bipartite Randomization

Instead of randomizing individual users, we identify isolated clusters and randomize at the cluster level.

<Callout type="example" title="Geographic clusters in a marketplace" collapsible defaultOpen={false}>

In a marketplace, we randomize entire cities: Chicago gets Treatment; Boston gets Control. Because a driver in Chicago cannot steal a ride from Boston, cannibalization is eliminated. The trade-off is a massive reduction in effective sample size — $N$ becomes the number of cities, not the number of users.

</Callout>

<Callout type="example" title="Graph cluster randomization in a social network" collapsible defaultOpen={false}>

Use a community detection algorithm (e.g., Louvain clustering) to partition the social graph into tight-knit friend groups. Randomize entire friend groups together. Most interactions then happen *within* the same treatment arm, preserving SUTVA within each cluster.

</Callout>

**Bipartite experiments:** In two-sided marketplaces where supply and demand cross heavily, we can randomize the *edges* of the bipartite graph (specific buyer-seller sessions) rather than the nodes. This requires complex inverse probability weighting (IPW) to estimate unbiased effects.

### 2. Switchback Testing

If geographic clusters are too few or too heterogeneous, we can randomize **time windows** within a single market.

A market alternates between Treatment and Control at regular intervals (e.g., 30-minute windows), randomly assigned. This eliminates user-level interference because the same users alternate between treatment states.

<Callout type="warning" title="Time-window autocorrelation" collapsible defaultOpen={false}>

Switchback designs introduce a new problem: time windows are heavily autocorrelated. High demand at 1:30 PM spills into the 2:00 PM window. Standard A/B variance formulas do not apply. We must use **Clustered Standard Errors** (clustering by day or week) or time-series modeling to estimate variance correctly.

</Callout>

### 3. Synthetic Control / Difference-in-Differences

Cluster randomization creates a massive statistical problem: it reduces the effective sample size to the number of clusters. If we're forced to test in just 1 Treatment city and 1 Control city, standard A/B testing is impossible.

In those cases, we turn to quasi-experimental methods:

**Difference-in-Differences (DiD):** Measure the parallel historical trends of both cities, then measure the post-intervention divergence. The treatment effect is the deviation from the pre-existing trend.

**Synthetic Control:** When no single city is a good match for the Treatment city, mathematically construct a "Synthetic Treatment City" — a weighted average of untreated cities that perfectly mirrored the Treatment city's pre-experiment metrics — and use that as the counterfactual.
